## Patterns for Reuse

### Pattern: Optional Dimension with Weight Normalization
- **When to use**: Adding opt-in capabilities to a scored multi-dimension system
- **Implementation**: Maintain two weight tables (with/without), select based on presence flag
- **Key**: `score: null` means "skip me in scoring", not "I scored zero"

### Pattern: Screenshot References Across 3 Export Formats
- **When to use**: Embedding visual evidence in HTML/PDF/MD reports
- **Implementation**:
  - HTML: `file:///absolute/path` with `<img>` tag
  - PDF: `pdf.image(path, x, w)` from fpdf2
  - MD: `![alt](./relative/path)` for portability
- **Key**: Copy screenshots to output dir alongside reports for self-contained exports

### Pattern: Agent-Dispatches-Agents Architecture
- **When to use**: When one dimension needs sub-tasks (e2e-validator dispatches browser-qa agents)
- **Implementation**: Top-level agent handles orchestration, sub-agents run in parallel via Task with run_in_background
- **Key**: The e2e-validator is the only dimension agent that itself dispatches sub-agents

## Recommendations for Similar Work

1. When adding dimensions to a multi-model review system, always handle the "only Claude can do X" asymmetry explicitly in dispatch, scoring, and display.
2. Screenshot embedding requires format-specific handling -- don't assume one approach works for all output formats.
3. Use `score: null` pattern for gracefully skipped dimensions rather than defaulting to a numeric value.
